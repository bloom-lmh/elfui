# ElfUI v0.1.0-beta.6 类型支持与单包入口缺陷计划

- 建档日期：2026-07-21
- 受影响版本：`v0.1.0-beta.5` 及更早版本
- 目标版本：`v0.1.0-beta.6`
- 审查范围：ElfUI 公共类型、宏编译器类型桥接、单包入口、ElfUI Kit 严格 TypeScript 消费

## 目标

本轮不以“让编辑器暂时消红”为完成标准，而是确保：

1. ElfUI 公共类型、宏编译器虚拟声明和实际运行时行为一致；
2. `defineEmits` 等宏 API 能自然接受有限键 `interface` / `type`；
3. 用户只安装并导入一个主包即可使用常规框架能力；
4. ElfUI Kit 同时通过仓库 TypeScript 与较新 DOM 类型检查；
5. 框架缺陷与 Kit 自身的业务类型错误分开修复和回归。

## 单包门面方案（已确认）

本轮采用与 Vue 相同的“用户单入口、内部继续分包”思路。“单包”指应用只声明一个直接运行时依赖并从一个入口导入日常 API，不指将 runtime、reactivity、compiler、router 物理合并成单体包。

目标用户用法：

```ts
import { defineHtml, html, onMount, useRef, useComputed, defineComponent } from "@elfui/core";
```

应用项目原则上只直接依赖：

```json
{
  "dependencies": {
    "@elfui/core": "^0.1.0-beta.6"
  },
  "devDependencies": {
    "@elfui/vite-plugin": "^0.1.0-beta.6"
  }
}
```

设计约束：

- `@elfui/core` 是唯一标准用户运行时入口，文档、模板和诊断统一使用这个包名；
- 主入口精选并重导出组件作者日常需要的 reactivity、runtime 和 macro 稳定 API，不使用无边界 `export *` 泄漏底层实现；
- 新增 `@elfui/core/internal`，仅承接编译产物 helper；编译器不得再让应用源码直接引用 `@elfui/runtime/internal`；
- `@elfui/runtime`、`@elfui/reactivity`、`@elfui/shared` 等包继续作为内部模块发布，供框架集成和高级扩展，但普通应用无需直接声明；
- `@elfui/compiler` 与 `@elfui/vite-plugin` 保持工具链包，Vite 插件作为开发依赖；router 等可选能力保持独立；
- 主入口不得把编译器或 Node 专用代码带进浏览器运行时；
- 必须用 exports、tree-shaking 和真实安装产物测试证明“全量入口方便但不强制全量打包”。

暂不把 npm alias `"elfui": "npm:@elfui/core@..."` 作为长期官方方案。若以后发布真正的 `elfui` 门面包，应只做稳定转发，并保证宏编译器、子路径 exports 和错误信息仍只有一个规范包名。

### 目标生成代码边界

```ts
// 用户源码与模型 helper
import { defineHtml, useModel } from "@elfui/core";

// 仅由编译器生成，用户不手写
import { attr, text, list } from "@elfui/core/internal";
```

应用的 `package.json` 不再需要直接声明 `@elfui/runtime` 或 `@elfui/reactivity`；它们仍可作为 `@elfui/core` 的传递依赖存在于安装目录中。

## `exactOptionalPropertyTypes` 规则

可选属性 `status?: StepStatus` 表示属性可以不存在，不表示属性存在时可以显式写入 `undefined`。错误形式：

```ts
return {
  status: child.status || undefined
};
```

推荐形式：

```ts
return {
  ...(child.status ? { status: child.status } : {})
};
```

如果业务语义确实需要“属性存在且值为 undefined”，才应将公共类型改为 `status?: StepStatus | undefined`；本例没有该需要。

## ElfUI Kit 审计结论

2026-07-21 使用 Kit 当前 TypeScript 配置执行基线扫描，共得到 145 条 raw `tsc` 诊断。其中：

- 86 条是宏模板中 `v-for` 局部变量或 `$event` 对 TypeScript 不可见造成的作用域诊断；这需要宏感知 language service / 转换后类型检查解决，不应靠全局声明掩盖；
- 框架根因类别为 `Ref<T> -> void` 5 条、有限键 emit interface 1 条、指令泛型 3 条、构造器包装类型 1 条；切换到本轮本地框架源码后四类均降为 0；
- AvatarGroup、Dropdown、Steps、PageHeader 四个目标组件使用本轮本地框架类型复查为 0 条诊断；
- 额外发现并修复 Dropdown 可选 modifiers 的不稳定窄化、虚拟引用的错误 `EventTarget` 断言，以及 Steps 对原生 `HTMLElement.title` 的错误可选覆盖。

QA-001 已在 2026-07-21 完成。Kit 新增 `scripts/typecheck-macros.mjs`：编译器负责模板表达式、`v-for`、`$event` 与组件契约诊断；TypeScript Program 对移除 `html` 模板字面量后的完整 setup 源码执行 `strict`、`noUncheckedIndexedAccess` 与 `exactOptionalPropertyTypes` 检查。该门禁覆盖 105 个宏组件，最终为 0 条宏诊断、0 条 TypeScript 诊断，并已加入 `build:lib` / `prepublishOnly`。

门禁落地过程中继续修复了 Kit 自身问题：模板 ref 错误保留 `.value`、事件表达式 DOM 类型过窄、可选属性显式写入 `undefined`、数组索引未判空、Table 列身份比较导致 `columnIndex = -1`、Splitter 默认值覆盖面板初始尺寸、TimePicker 范围值退化为普通数组、Sticky 的 `Window | HTMLElement` 窄化，以及菜单、表单与指令的严格类型缺陷。

## API 兼容说明

`Ref.set()`、可写 computed 的 `set()` 与 `ModelRef.set()` 从链式返回对象调整为 `void`。普通调用不变：

```ts
active.set(next);
```

唯一需要迁移的是未公开承诺且与常规响应式 API 不一致的链式写法：

```ts
// 旧写法
active.set(first).set(second);

// 新写法
active.set(first);
active.set(second);
```

- 公共类型测试覆盖 `Ref.set`、有限键事件接口、typed runtime emit names 和非法事件名；
- ElfUI Kit 不再依赖编辑器版本差异隐藏类型问题；
- 普通应用只直接安装 `@elfui/core` 即可使用日常 API，宏生成代码不越过该直接依赖边界；
- 单包入口通过真实 npm tarball 消费、ESM、类型、exports 和 tree-shaking 验证；
- 本文档所有条目有明确的“已修复 / 延后及原因 / Kit 自身问题”结论。
